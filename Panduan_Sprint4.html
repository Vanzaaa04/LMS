👤 TUGAS TIO - Lab CRUD (Praktikum)

Branch: fitur/backend-tio-lab

Deskripsi

Praktikum (Lab) adalah sesi praktik mandiri yang Dosen berikan kepada Mahasiswa di dalam mata kuliah. Dosen menulis instruksi langkah-langkah praktikum, lalu Mahasiswa membaca dan mengerjakannya secara mandiri. Tugas kamu di Sprint ini adalah membangun seluruh API manajemen praktikum dari nol: mulai dari pembuatan sesi baru, daftar praktikum, detail instruksi, sampai pengeditan dan penghapusan. Kamu juga bertanggung jawab menyiapkan tabel LabSubmission di database agar Dimas bisa langsung menggunakannya untuk fitur pengumpulan hasil praktikum. API yang kamu buat nanti akan dipakai oleh Frontend untuk merender halaman daftar praktikum, halaman detail instruksi, dan form kelola praktikum oleh dosen.

B. Pembaruan Database (Prisma)

  1. Buka file skema database di apps/api/prisma/schema.prisma.
  2. Model PracticalLab sudah ada di sana, kamu tidak perlu membuatnya lagi. Tapi kamu perlu menambahkan model baru bernama LabSubmission. Model ini fungsinya sama seperti AssignmentSubmission yang sudah ada, yaitu sebagai tempat menyimpan data pengumpulan hasil praktikum mahasiswa. Kamu bisa melihat struktur AssignmentSubmission di file yang sama sebagai referensi polanya.
  3. Di dalam model LabSubmission, pastikan ada: field ID (UUID), field labId yang merujuk ke PracticalLab, field studentId yang merujuk ke User, field fileUrl dan note untuk menyimpan hasil kerja mahasiswa, field status bertipe String (default "PENDING", nanti bisa berubah jadi "GRADED"), field score (angka) dan feedback (teks) untuk penilaian dosen, serta timestamp createdAt dan updatedAt.
  4. Tambahkan aturan unik (unique constraint) kombinasi antara labId dan studentId agar satu mahasiswa tidak bisa submit dua kali di lab yang sama.
  5. Pada kedua relasi (lab dan student), pasang argumen onDelete: Cascade agar saat Lab atau User dihapus, data submission-nya otomatis ikut terhapus bersih.
  6. Jangan lupa tambahkan relasi balikan (array) di model PracticalLab dan model User, supaya Prisma bisa mengenali hubungan dua arah.
  7. Jalankan migrasi database:
    8. npx prisma migrate dev --name add_lab_submission_model
npx prisma generate

C. Mulai Coding

Kamu perlu membuat modul baru di dalam folder apps/api/src/lab/. Gunakan generator NestJS untuk membuatnya secara otomatis (jalankan dari folder apps/api): generate module, controller, dan service untuk lab. Pastikan PrismaService terdaftar di providers modul tersebut.

  1. Buka file service (lab.service.ts). Import PrismaService dan masukkan ke constructor. Kamu harus membuat 5 fungsi:
    • Fungsi create - Menerima 3 data: judul lab (title), instruksi panjang langkah-langkah praktikum (instructions), dan ID course-nya (courseId). Sebelum menyimpan, cek dulu apakah course-nya ada di database. Kalau tidak ada, kasih error NotFoundException. Lalu cek apakah user yang melakukan request adalah dosen pemilik course tersebut (bandingkan course.instructorId dengan userId dari token JWT). Kalau bukan pemilik, tolak dengan ForbiddenException. Kalau semua lolos, simpan data lab baru ke tabel PracticalLab.
    • Fungsi findAll - Ambil semua data lab dari database. Sediakan opsi filter berdasarkan parameter URL courseId (contoh: ?courseId=xxx), agar frontend bisa menampilkan daftar praktikum untuk mata kuliah tertentu saja. Return array berupa id, judul, dan tanggal pembuatan lab.
    • Fungsi findOne - Ambil detail satu lab berdasarkan ID. Tujuan endpoint ini adalah agar mahasiswa bisa membaca panduan teknis praktikum. Return judul dan teks instruksi tata cara praktikum secara utuh. Kalau ID tidak ditemukan, kasih error NotFoundException.
    • Fungsi update - Menerima perubahan judul dan/atau instruksi lab. Sebelum mengeksekusi update, wajib validasi bahwa user yang melakukan request adalah dosen pemilik course (sama seperti fungsi create). Kalau bukan, tolak dengan ForbiddenException.
    • Fungsi remove - Hapus lab berdasarkan ID. Validasi kepemilikan dosen terlebih dahulu. Karena kamu sudah memasang onDelete: Cascade di model LabSubmission, saat lab dihapus maka seluruh riwayat pengumpulan tugas mahasiswa terkait lab tersebut akan ikut lenyap otomatis, kamu tidak perlu menghapusnya satu per satu.
  2. Buka file controller (lab.controller.ts). Masukkan LabService ke constructor. Buat 5 route endpoint:
    • POST /labs - Buat praktikum baru. Pasang guard JWT agar hanya user yang sudah login yang bisa mengakses.
    • GET /labs - Daftar praktikum. Terima query parameter courseId yang bersifat opsional.
    • GET /labs/:id - Detail instruksi praktikum.
    • PUT /labs/:id - Edit praktikum. Pasang guard JWT.
    • DELETE /labs/:id - Hapus praktikum. Pasang guard JWT.
  3. Pastikan modul terdaftar. Buka apps/api/src/app.module.ts. Jika kamu menggunakan generator NestJS, modul biasanya sudah otomatis terdaftar. Kalau belum, tambahkan LabModule ke dalam array imports.

Di Sprint sebelumnya, seluruh endpoint Course dan Material sudah dibangun (termasuk endpoint GET /courses/my untuk menampilkan daftar matkul yang diikuti mahasiswa). Tugas kamu di Sprint ini adalah melakukan pengujian menyeluruh terhadap semua endpoint buatanmu tersebut menggunakan aplikasi Postman. Kamu bertanggung jawab memastikan bahwa setiap API benar-benar berfungsi sesuai spesifikasi, response JSON-nya rapi dan konsisten, serta pesan error HTTP (seperti 400, 403, 404) dikeluarkan dengan tepat ketika ada validasi yang gagal. Jika ditemukan bug apapun, kamu yang memperbaikinya langsung di kode, lalu dokumentasikan hasilnya. Hasil testing-mu akan dipakai Leader sebagai acuan apakah modul Course & Material siap dikoneksikan ke frontend.

B. Skenario Testing Course

Buka Postman dan uji seluruh endpoint Course satu per satu. Lakukan simulasi lengkap mengikuti alur ini:

  1. Register akun Dosen dan akun Mahasiswa lewat POST /auth/register. Login keduanya untuk mendapatkan token masing-masing.
  2. Buat Mata Kuliah baru lewat POST /courses menggunakan token Dosen. Pastikan berhasil (status 201) dan response menampilkan data course yang dibuat.
  3. Lihat daftar semua matkul lewat GET /courses. Pastikan course yang baru dibuat muncul di daftar dan ada informasi nama dosen pengampu.
  4. Lihat detail satu matkul lewat GET /courses/[ID course]. Pastikan response memuat seluruh data relasi: materials, quizzes, assignments, dan labs (walau masih kosong, field-nya harus ada berupa array kosong).
  5. Enroll Mahasiswa ke matkul lewat POST /courses/[ID course]/enroll menggunakan token Mahasiswa. Pastikan berhasil. Lalu coba enroll lagi, pastikan ditolak dengan error 400 karena sudah terdaftar.
  6. Panggil GET /courses/my menggunakan token Mahasiswa. Pastikan course yang baru di-enroll muncul di daftar kelasku.
  7. Edit matkul lewat PUT /courses/[ID course] menggunakan token Dosen pemilik. Pastikan berhasil. Lalu coba edit menggunakan token dosen lain — pastikan ditolak dengan error 403 Forbidden.
  8. Hapus matkul lewat DELETE /courses/[ID course]. Pastikan berhasil dan mekanisme Cascade Delete bekerja, yaitu semua materi, kuis, tugas, lab, dan data enrollment yang terkait ikut terhapus dari database.

C. Skenario Testing Material

Buat lagi course baru untuk testing Material. Lalu uji seluruh endpoint Material:

  1. Upload materi tipe TEXT lewat POST /materials. Isi: judul, type "TEXT", konten teks, dan courseId. Pastikan berhasil.
  2. Upload materi tipe VIDEO lewat POST /materials. Isi: judul, type "VIDEO", dan URL video. Pastikan berhasil.
  3. Upload materi tipe DOCUMENT lewat POST /materials. Isi: judul, type "DOCUMENT", dan URL file. Pastikan berhasil.
  4. Lihat detail satu materi lewat GET /materials/[ID materi]. Pastikan response menampilkan judul, tipe, dan konten/URL secara utuh.
  5. Edit materi lewat PUT /materials/[ID materi] menggunakan token Dosen pemilik. Pastikan data berubah. Lalu coba edit menggunakan token dosen lain, pastikan error 403.
  6. Hapus materi lewat DELETE /materials/[ID materi] menggunakan token Dosen pemilik. Pastikan data terhapus dari database.

D. Bug Fixing

Jika ditemukan error/bug saat testing, buka file service yang relevan di folder apps/api/src/course/ atau apps/api/src/material/. Perbaiki masalahnya langsung. Fokus pada: response JSON yang tidak rapi (field yang seharusnya tidak muncul), pesan error HTTP yang salah kode-nya (misal seharusnya 404 tapi malah 500), dan validasi yang terlewat.

E. Dokumentasi Hasil

Buat file baru apps/api/test/TESTING_RESULTS.md. Di dalamnya, catat hasil testing kamu dalam bentuk tabel: nomor, endpoint, skenario, status (PASS/FAIL), dan catatan jika ada bug yang diperbaiki. File ini yang akan di-review oleh Leader.

C. Mulai Coding

Buat modul baru di folder apps/api/src/lab-submission/. Gunakan generator NestJS (jalankan dari folder apps/api) untuk membuat module, controller, dan service untuk lab-submission. Pastikan PrismaService terdaftar di providers modul tersebut. Kamu bisa lihat modul assignment di folder apps/api/src/assignment/ sebagai referensi pola kode — konsepnya sangat mirip.

  1. Buka file service (lab-submission.service.ts). Import PrismaService dan masukkan ke constructor. Kamu harus membuat 3 fungsi utama:
    • Fungsi submit - Menerima ID lab (dari parameter URL), userId (dari token JWT), fileUrl (link file hasil praktikum), dan note (catatan tambahan, opsional). Lakukan 3 validasi berturut-turut sebelum menyimpan:
      - Validasi 1: Cek lab-nya ada di database. Kalau tidak ada, kasih error NotFoundException.
      - Validasi 2: Cek mahasiswa terdaftar di kelas lab tersebut. Query ke tabel Enrollment menggunakan kombinasi userId dan courseId milik lab. Kalau tidak terdaftar, tolak dengan ForbiddenException. Ini mencegah penyusup yang bukan anggota kelas mengumpulkan hasil.
      - Validasi 3: Cek apakah mahasiswa sudah pernah submit sebelumnya untuk lab ini. Query ke tabel LabSubmission menggunakan kombinasi labId dan studentId. Kalau sudah ada, tolak dengan BadRequestException. Ini mencegah pengumpulan ganda.
      Kalau ketiga validasi lolos, simpan data submission baru ke tabel LabSubmission dengan status awal "PENDING".
    • Fungsi getSubmissions - Menerima ID lab dan userId (dosen). Tujuannya agar dosen bisa melihat kompilasi siapa saja yang sudah mengumpulkan hasil praktikum. Cek dulu bahwa lab-nya ada dan userId adalah dosen pemilik course. Lalu query seluruh data submission untuk lab tersebut, termasuk nama lengkap dan email mahasiswanya (gunakan relasi include ke tabel User). Return array berupa nama mahasiswa, tautan file, stempel waktu pengumpulan, status penilaian (PENDING/GRADED), dan skor jika sudah dinilai.
    • Fungsi grade - Menerima ID submission, userId (dosen), score (angka 0-100), dan feedback (teks komentar evaluatif, opsional). Cek submission ada di database. Cek bahwa userId adalah dosen pemilik course dari lab tersebut (ini membutuhkan query relasi berantai: submission > lab > course > instructorId). Update data submission: simpan nilai score, simpan teks feedback, dan ubah status dari "PENDING" menjadi "GRADED". Setelah disimpan, mahasiswa akan bisa melihat nilainya secara transparan.
  2. Buka file controller (lab-submission.controller.ts). Karena endpoint-mu memiliki awalan URL yang berbeda-beda (/labs/... untuk submit dan lihat submission, serta /lab-submissions/... untuk grading), atur dekorator @Controller() tanpa prefix agar kamu bisa menulis path lengkap di setiap route. Buat 3 route:
    • POST /labs/:id/submit - Mahasiswa kumpul hasil lab. Pasang guard JWT.
    • GET /labs/:id/submissions - Dosen lihat daftar submission. Pasang guard JWT.
    • PUT /lab-submissions/:id/grade - Dosen kasih nilai. Pasang guard JWT.
  3. Pastikan modul terdaftar. Cek apps/api/src/app.module.ts, pastikan modul LabSubmissionModule sudah ada di imports.
